1. 온프레미스 vs 클라우드 보안 차이

1.1 온프레미스 보안의 전통적 접근 방식

전통적인 온프레미스(On-Premises) 환경에서는 기업이 데이터센터의 모든 계층에 대한 보안 책임을 집니다. 물리적 보안(출입 통제, CCTV, 잠금장치)부터 네트워크 장비(방화벽, IDS/IPS, 스위치), 서버 하드웨어, 운영체제 패치, 미들웨어, 애플리케이션 코드, 그리고 데이터에 이르기까지 전부 직접 관리해야 합니다.

비유: 자가 주택 vs 아파트

온프레미스 보안은 단독 주택을 소유한 것과 같습니다. 지붕 수리, 정원 관리, 담장 설치, 도어록 교체 등 집의 모든 것을 직접 책임져야 합니다. 반면 클라우드는 아파트에 사는 것과 비슷합니다. 건물 구조, 엘리베이터, 공용 시설은 관리사무소(AWS)가 관리하고, 세대 내부의 인테리어와 귀중품 관리는 입주자(고객)가 책임집니다.

온프레미스 환경의 보안 특징을 정리하면 다음과 같습니다:

  • 물리적 경계가 명확: 데이터센터 건물 자체가 보안 경계. 출입 통제로 1차 방어
  • 네트워크 경계(Perimeter) 중심: 방화벽이 내부/외부를 명확히 구분. "성벽 안은 안전하다"는 개념
  • 하드웨어 직접 통제: 서버, 스토리지, 네트워크 장비를 물리적으로 소유하고 관리
  • 변경 속도가 느림: 보안 장비 도입에 수주~수개월 소요. 조달, 설치, 설정 과정 필요
  • 확장성 제한: 보안 인프라를 확장하려면 추가 하드웨어 구매 필요

1.2 클라우드 환경에서 달라지는 점

클라우드로 전환하면 보안 패러다임이 근본적으로 변합니다. 더 이상 물리적 서버를 만질 수 없고, 네트워크 경계가 사라지며, 모든 것이 API를 통해 관리됩니다.

경계의 소멸 (Dissolving Perimeters)

온프레미스에서는 방화벽 안쪽이 "신뢰 영역"이었습니다. 하지만 클라우드에서는 어디서든 인터넷을 통해 리소스에 접근할 수 있으므로, 전통적인 네트워크 경계 개념이 무의미해집니다. 대신 자격증명(Identity)이 새로운 경계가 됩니다.

핵심 개념: "Identity is the New Perimeter"

클라우드에서는 누가(Who) 접근하느냐가 가장 중요한 보안 경계입니다. IAM 사용자, 역할, 정책이 방화벽의 역할을 대체합니다. 유출된 Access Key 하나로 전체 인프라가 위험에 처할 수 있습니다.

공유 인프라 (Shared Infrastructure)

퍼블릭 클라우드에서는 여러 고객의 워크로드가 동일한 물리적 하드웨어 위에서 실행됩니다. AWS는 하이퍼바이저 수준의 격리를 통해 테넌트 간 분리를 보장하지만, 고객은 자신의 영역에서 적절한 보안 설정을 해야 합니다.

API 기반 관리

클라우드의 모든 리소스는 API를 통해 생성, 수정, 삭제됩니다. 이는 양날의 검입니다:

  • 장점: 보안 정책을 코드로 정의(Infrastructure as Code)하여 일관성 있게 적용 가능
  • 위험: 잘못된 API 호출 하나로 S3 버킷을 퍼블릭으로 열거나, 보안그룹에 0.0.0.0/0을 추가할 수 있음

1.3 클라우드 보안의 장점과 새로운 과제

2024년 기준, 클라우드 보안 사고의 약 80% 이상이 고객 측 설정 오류에서 발생합니다. S3 버킷 퍼블릭 노출, 보안그룹 전체 개방, IAM 과도 권한 등이 대표적입니다. 이는 AWS의 보안 결함이 아니라, 공동 책임 모델에서 고객 영역의 관리 부재가 원인입니다.

ShopEasy와의 연결

기초 과정에서 구축한 ShopEasy는 빠른 배포에 초점을 맞추어 보안 설정이 최소화되어 있습니다. 이 과정을 통해 ShopEasy에 체계적인 보안을 적용하면서, 클라우드 보안의 핵심 원칙을 실습합니다.

2. AWS 공동 책임 모델

2.1 Shared Responsibility Model이란?

AWS 공동 책임 모델(Shared Responsibility Model)은 클라우드 보안의 가장 기본적이면서도 중요한 개념입니다. 한마디로 요약하면: "AWS는 클라우드 자체의 보안을, 고객은 클라우드 안에서의 보안을 책임진다"입니다.

Security OF the Cloud vs Security IN the Cloud

Security OF the Cloud (AWS 책임): 클라우드 인프라 자체를 보호합니다. 글로벌 데이터센터, 하드웨어, 네트워킹 장비, 하이퍼바이저, 호스트 OS를 AWS가 관리합니다.

Security IN the Cloud (고객 책임): 클라우드 위에서 실행하는 모든 것을 보호합니다. 게스트 OS 패치, 애플리케이션 코드, 데이터 암호화, IAM 설정, 네트워크 설정(보안그룹, NACL)이 여기에 해당합니다.

2.2 AWS가 책임지는 영역

AWS는 다음과 같은 영역의 보안을 책임지고 관리합니다:

  • 물리적 보안: 데이터센터 건물, 출입 통제, 환경 제어(온도, 습도), 전력 공급, 재해 대비
  • 하드웨어/글로벌 인프라: 서버, 스토리지, 네트워킹 장비의 유지보수와 교체
  • 호스트 운영체제: 하이퍼바이저 및 호스트 OS 패치와 보안 업데이트
  • 네트워킹 인프라: 리전, 가용 영역, 엣지 로케이션 간 연결
  • 관리형 서비스 인프라: RDS, DynamoDB, Lambda 등의 기반 인프라 운영
  • 컴플라이언스: SOC 1/2/3, PCI DSS, ISO 27001/27017/27018, HIPAA 등 국제 인증 획득 및 유지

2.3 고객이 책임지는 영역

고객은 클라우드에서 사용하는 모든 서비스와 데이터의 보안을 직접 관리해야 합니다:

  • 데이터 관리: 데이터 분류, 암호화, 백업, 접근 제어
  • 자격증명 & 접근 제어: IAM 사용자/역할/정책 관리, MFA 설정, 비밀번호 정책
  • 운영체제 & 네트워크: EC2 인스턴스의 게스트 OS 패치, 보안그룹/NACL 설정
  • 애플리케이션 보안: 코드 보안, 입력 검증, 인증/인가 로직
  • 네트워크 트래픽 보호: SSL/TLS 적용, VPN 설정, 전송 중 데이터 암호화
  • 로깅 & 모니터링: CloudTrail 활성화, 로그 분석, 알림 설정

2.5 실제 사례로 이해하기

사례 1: S3 버킷 퍼블릭 노출 사고

2017년, 미국의 한 대형 통신사가 AWS S3 버킷의 접근 정책을 잘못 설정하여 약 1억 4천만 명의 개인정보가 노출되었습니다. S3 서비스 자체에는 문제가 없었으며, 고객이 버킷 정책을 퍼블릭으로 설정한 것이 원인이었습니다. 이는 명확히 "Security IN the Cloud" (고객 책임) 영역입니다.

사례 2: Access Key 유출

개발자가 GitHub 공개 리포지토리에 AWS Access Key를 실수로 커밋했습니다. 수 분 내에 봇이 해당 키를 탐지하여 대량의 EC2 인스턴스를 암호화폐 채굴에 사용했습니다. 하루 만에 수만 달러의 청구가 발생했습니다. 이 역시 고객 책임 영역입니다. 해결책: IAM 역할 사용, Access Key 대신 임시 자격증명 활용, git-secrets 도구 적용.

사례 3: AWS의 책임 이행

2019년 AWS가 하이퍼바이저 취약점(Nitro 관련)을 발견했을 때, 고객에게 영향 없이 인프라를 패치했습니다. 고객은 어떤 조치도 취할 필요가 없었으며, 이는 "Security OF the Cloud" (AWS 책임) 영역이기 때문입니다.

3. 서비스별 책임 범위

AWS의 모든 서비스가 동일한 책임 구조를 갖는 것은 아닙니다. 서비스 유형에 따라 고객과 AWS의 책임 범위가 달라집니다. 일반적으로 IaaS는 고객 책임이 많고, SaaS로 갈수록 AWS 책임이 커집니다.

3.1 IaaS (Infrastructure as a Service) - EC2

EC2는 가장 기본적인 IaaS 서비스로, 고객에게 가장 많은 제어권과 동시에 가장 많은 보안 책임을 부여합니다.

  • 고객 책임: 게스트 OS 선택/패치, 미들웨어 설치/설정, 애플리케이션 배포/보안, 보안그룹 설정, 데이터 암호화, 로깅 설정
  • AWS 책임: 물리 서버, 네트워크 인프라, 하이퍼바이저(Nitro), 호스트 OS
EC2에서 자주 발생하는 보안 실수

1) OS 패치를 방치하여 알려진 취약점 노출
2) 보안그룹에서 SSH(22번 포트)를 0.0.0.0/0으로 전체 개방
3) 인스턴스에 Access Key를 하드코딩
4) 퍼블릭 서브넷에 불필요한 인스턴스 배치

3.2 PaaS (Platform as a Service) - RDS, Lambda

관리형 서비스(Managed Service)를 사용하면 AWS가 더 많은 인프라를 관리합니다. 고객은 서비스 설정과 데이터에만 집중할 수 있습니다.

RDS (Relational Database Service)

  • AWS 관리: DB 엔진 패치(선택 가능), 백업 인프라, 하드웨어 유지보수, OS 패치
  • 고객 책임: DB 사용자/비밀번호 관리, 보안그룹 설정, 암호화 활성화 여부, 접근 권한 관리, 데이터 분류

Lambda (Serverless)

  • AWS 관리: 실행 환경, 런타임 보안, OS, 확장성, 가용성
  • 고객 책임: 함수 코드 보안, IAM 실행 역할 설정, 환경변수 암호화, VPC 설정(필요시)

3.3 SaaS (Software as a Service) - S3 정적 호스팅

S3와 같은 완전 관리형 서비스에서는 AWS가 대부분의 인프라와 서비스 운영을 담당합니다. 고객은 데이터와 접근 제어에만 집중하면 됩니다.

  • AWS 관리: 스토리지 인프라, 내구성(11 nines), 가용성, 서비스 운영 전체
  • 고객 책임: 버킷 정책, 퍼블릭 액세스 차단 설정, 데이터 암호화, 객체 ACL, 버전 관리

EC2에 직접 MySQL을 설치하는 것보다 RDS를 사용하면, OS 패치와 DB 엔진 업데이트를 AWS에 위임할 수 있습니다. 서버리스(Lambda)를 사용하면 서버 관리 자체가 불필요합니다. "관리형 서비스 = 보안 책임 범위 축소"라는 공식을 기억하세요.

4. 클라우드 보안의 핵심 원칙

클라우드 보안을 효과적으로 구현하기 위한 4가지 핵심 원칙을 알아봅니다. 이 원칙들은 AWS Well-Architected Framework의 보안 필러(Security Pillar)에도 반영되어 있으며, 이 과정 전체를 관통하는 기본 철학입니다.

4.1 최소 권한 원칙 (Principle of Least Privilege)

정의

사용자, 서비스, 애플리케이션에 업무 수행에 꼭 필요한 최소한의 권한만 부여하는 원칙입니다. "만약을 위해 넓은 권한을 주자"는 접근은 절대 하지 않습니다.

비유: 호텔 키 카드

호텔에서 투숙객에게 모든 방의 마스터키를 주지 않습니다. 본인 객실의 키만 줍니다. 수영장을 이용하려면 별도의 접근 권한이 필요합니다. AWS에서도 마찬가지로, S3에만 접근하면 되는 서비스에 EC2 관리 권한까지 줄 필요가 없습니다.

최소 권한 원칙의 실무 적용:

  • 와일드카드(*) 지양: "Action": "s3:*" 대신 "Action": "s3:GetObject"처럼 구체적으로
  • 리소스 범위 한정: "Resource": "*" 대신 특정 ARN 지정
  • 정기적 검토: IAM Access Analyzer로 미사용 권한 탐지 및 제거
  • 임시 자격증명 사용: 영구 Access Key 대신 IAM 역할과 STS 활용
json 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadProductImages",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::shopeasy-product-images",
        "arn:aws:s3:::shopeasy-product-images/*"
      ]
    }
  ]
}

위 정책은 ShopEasy의 상품 이미지 버킷에 대해 읽기 전용 접근만 허용하는 최소 권한 정책의 예시입니다.

4.2 심층 방어 (Defense in Depth)

정의

단일 보안 계층에 의존하지 않고, 여러 겹의 보안 레이어를 중첩하여 방어하는 전략입니다. 한 계층이 뚫려도 다음 계층에서 방어할 수 있습니다.

비유: 중세 성의 방어

중세 성은 해자(물웅덩이) → 성벽 → 내성 → 탑 → 금고의 다층 방어를 갖추었습니다. 적이 해자를 건너도 성벽이 있고, 성벽을 넘어도 내성이 있습니다. AWS에서도 마찬가지로 WAF → 보안그룹 → NACL → OS 방화벽 → 애플리케이션 인증의 다층 방어를 구성합니다.

ShopEasy 심층 방어 (Defense in Depth) 계층
Layer 1: 엣지 보안
CloudFront + AWS WAF + Shield
↓
Layer 2: 네트워크 보안
VPC + NACL + 보안그룹 + 프라이빗 서브넷
↓
Layer 3: 자격증명 & 접근제어
IAM 역할 + 정책 + MFA + STS
↓
Layer 4: 호스트 보안
OS 패치 + Inspector + SSM
↓
Layer 5: 데이터 보호
KMS + S3 암호화 + RDS 암호화 + Secrets Manager
↓
Layer 6: 탐지 & 모니터링 (모든 계층을 감시)
CloudTrail + GuardDuty + Config + Security Hub + CloudWatch

4.3 제로 트러스트 (Zero Trust)

정의

"절대 신뢰하지 말고, 항상 검증하라 (Never Trust, Always Verify)"
네트워크 내부에 있든 외부에 있든, 모든 접근 요청을 의심하고 인증/인가를 수행하는 보안 모델입니다.

제로 트러스트의 AWS 적용 방법:

제로 트러스트 원칙 AWS 구현 방법 ShopEasy 적용 예
모든 요청 인증 IAM 역할 + STS 임시 자격증명 EC2가 S3 접근 시 인스턴스 프로파일 사용
최소 권한 접근 세분화된 IAM 정책 API 서버에 필요한 S3, RDS 접근만 허용
마이크로 세그멘테이션 보안그룹 + 프라이빗 서브넷 DB는 프라이빗 서브넷, API 서버만 접근 허용
지속적 모니터링 CloudTrail + GuardDuty 모든 API 호출 로깅, 이상 행위 탐지
전송 암호화 TLS/SSL 강제 ALB-EC2, EC2-RDS 간 통신 암호화

4.4 자동화된 보안 (Security Automation)

정의

보안 정책, 감사, 대응을 수작업이 아닌 자동화로 처리하는 원칙입니다. 사람은 실수하지만, 코드는 일관되게 실행됩니다. 클라우드의 API 기반 특성을 활용하여 보안을 코드로 정의하고 자동화합니다.

자동화 가능한 보안 영역:

  • 인프라 코드화 (IaC): CloudFormation, Terraform으로 보안 설정을 코드로 관리
  • 컴플라이언스 자동 검사: AWS Config Rules로 설정 위반을 자동 탐지
  • 사고 자동 대응: EventBridge + Lambda로 보안 이벤트에 자동 대응
  • 패치 자동화: Systems Manager로 OS 패치를 자동 적용
  • 비밀 자동 교체: Secrets Manager로 DB 비밀번호 자동 로테이션
python 
# 보안 자동화 예시: 퍼블릭 S3 버킷 자동 차단
# EventBridge 트리거 → Lambda 실행

import boto3

def lambda_handler(event, context):
    s3 = boto3.client('s3')
    bucket_name = event['detail']['requestParameters']['bucketName']

    # 퍼블릭 액세스 차단 설정 적용
    s3.put_public_access_block(
        Bucket=bucket_name,
        PublicAccessBlockConfiguration={
            'BlockPublicAcls': True,
            'IgnorePublicAcls': True,
            'BlockPublicPolicy': True,
            'RestrictPublicBuckets': True
        }
    )

    print(f"퍼블릭 액세스 차단 완료: {bucket_name}")
    return {'statusCode': 200}

Lab: 클라우드 보안 원칙 셀프 체크

현재 ShopEasy 아키텍처를 4가지 보안 원칙 관점에서 진단해봅니다.

ShopEasy 현재 상태 확인

AWS 콘솔에 로그인하여 기초 과정에서 구축한 ShopEasy의 현재 리소스를 확인합니다. EC2, RDS, S3, ALB, VPC 등 각 서비스의 현재 보안 설정 상태를 메모합니다.

최소 권한 원칙 점검

IAM 콘솔로 이동하여 현재 사용자와 역할의 정책을 확인합니다. AdministratorAccess 같은 과도한 권한이 부여되어 있지는 않은지, EC2에 인스턴스 프로파일이 연결되어 있는지 확인합니다.

심층 방어 점검

VPC → 보안그룹에서 인바운드 규칙을 확인합니다. 0.0.0.0/0으로 열려 있는 포트가 있는지, DB가 퍼블릭 서브넷에 있지는 않은지 체크합니다. WAF가 적용되어 있는지 확인합니다.

제로 트러스트 점검

EC2 인스턴스에 Access Key가 하드코딩되어 있지는 않은지 확인합니다. ~/.aws/credentials 파일이나 환경변수에 영구 키가 설정되어 있는지 체크합니다. RDS 비밀번호가 코드에 하드코딩되어 있는지 확인합니다.

자동화 점검

CloudTrail이 활성화되어 있는지 확인합니다. AWS Config가 설정되어 있는지, GuardDuty가 켜져 있는지 체크합니다. 대부분의 항목이 미설정 상태일 것입니다. 이것이 이 과정에서 해결할 과제입니다.

보안 진단 결과 정리

각 원칙별로 현재 상태를 "적용됨 / 미적용 / 부분 적용"으로 분류합니다. 이 결과가 이 과정의 보안 적용 로드맵이 됩니다. Chapter 02에서 이 진단 결과를 기반으로 구체적인 보안 서비스 매핑을 수행합니다.

AWS Well-Architected Framework - 보안 필러

AWS는 Well-Architected Framework에서 보안을 6가지 영역으로 정리합니다:
1) 보안 기반: 계정 관리, 컨트롤 타워
2) 자격증명 & 접근 관리: IAM, 인증, 인가
3) 탐지: 로깅, 모니터링, 이상 탐지
4) 인프라 보호: VPC, 네트워크, 호스트
5) 데이터 보호: 암호화, 키 관리, 데이터 분류
6) 사고 대응: 대응 계획, 자동화, 사후 분석

이 과정은 위 6가지 영역을 모두 다루며, ShopEasy에 실습으로 적용합니다.
