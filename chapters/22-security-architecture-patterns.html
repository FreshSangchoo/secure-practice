1. Well-Architected Framework 보안 필러 6가지 원칙

AWS Well-Architected Framework은 클라우드 아키텍처를 평가하고 개선하기 위한 모범 사례 프레임워크입니다. 보안 필러(Security Pillar)는 6가지 설계 원칙을 제시하며, 이 과정에서 학습한 모든 내용이 이 원칙들에 매핑됩니다.

과정 총정리

이 섹션은 22개 챕터에서 배운 내용을 Well-Architected 보안 필러의 관점에서 정리합니다. 각 원칙이 ShopEasy에 어떻게 적용되었는지 되돌아보며, 전체 보안 아키텍처의 큰 그림을 완성합니다.

원칙 1: 강력한 자격 증명 기반 구현 (Implement a Strong Identity Foundation)

최소 권한 원칙을 적용하고, 역할 분리를 통해 각 AWS 리소스에 대한 접근을 세밀하게 통제합니다. 장기 자격 증명 대신 임시 자격 증명을 사용하는 것이 핵심입니다.

실천 항목 관련 챕터 ShopEasy 적용
최소 권한 원칙 Ch.03 IAM 정책 심화, Ch.05 권한 경계 개발자/운영자 역할별 세분화된 IAM 정책
임시 자격 증명 사용 Ch.04 IAM 역할 & STS EC2 인스턴스 프로파일, AssumeRole 기반 접근
MFA 강제 Ch.06 이커머스 IAM 실습 IAM 사용자 MFA 필수, 민감 작업에 MFA 조건
권한 경계 설정 Ch.05 Permission Boundary 위임된 관리자의 권한 상한 제한

원칙 2: 추적 가능성 활성화 (Enable Traceability)

모든 환경 변경과 작업을 실시간으로 모니터링하고 기록합니다. 로그와 메트릭을 자동으로 수집하여, 보안 사고 시 빠르게 원인을 파악할 수 있어야 합니다.

실천 항목 관련 챕터 ShopEasy 적용
API 호출 기록 Ch.17 CloudTrail 모든 리전 CloudTrail 활성화, S3 저장
구성 변경 추적 Ch.18 AWS Config 리소스 구성 변경 이력 기록 및 규칙 평가
위협 탐지 Ch.19 GuardDuty & Security Hub GuardDuty 활성화, Security Hub 중앙 대시보드
네트워크 트래픽 기록 Ch.07 VPC 보안 심화 VPC Flow Logs 활성화 및 분석

원칙 3: 모든 계층에 보안 적용 (Apply Security at All Layers)

단일 보안 메커니즘에 의존하지 않고, 네트워크, 호스트, 애플리케이션, 데이터 등 모든 계층에 보안 통제를 적용하는 심층 방어(Defense in Depth) 전략입니다.

실천 항목 관련 챕터 ShopEasy 적용
네트워크 보안 Ch.07 VPC 보안, Ch.09 네트워크 보안 실습 보안그룹 체이닝, NACL, 프라이빗 서브넷
엣지 보안 Ch.08 WAF, Ch.14 CloudFront 보안 WAF 규칙, CloudFront OAC, HTTPS 강제
애플리케이션 보안 Ch.15 API Gateway, Ch.16 Lambda 보안 API 인증, 스로틀링, Lambda 최소 권한
데이터 보안 Ch.10-13 데이터 보호 섹션 KMS 암호화, S3 보안, Secrets Manager

원칙 4: 보안 모범 사례 자동화 (Automate Security Best Practices)

보안 통제를 코드로 정의하고, 자동으로 적용합니다. 수동 작업은 실수와 지연을 초래하므로, 가능한 한 모든 보안 절차를 자동화합니다.

실천 항목 관련 챕터 ShopEasy 적용
사고 대응 자동화 Ch.20 보안 사고 대응 자동화 EventBridge + Lambda 자동 격리
정책 자동 적용 Ch.21 Organizations & SCP SCP로 전체 계정 정책 자동 적용
비밀 자동 로테이션 Ch.13 Secrets Manager DB 비밀번호 자동 로테이션
컴플라이언스 자동 평가 Ch.18 AWS Config Config Rules로 지속적 컴플라이언스 평가

원칙 5: 전송 중 및 저장 중 데이터 보호 (Protect Data in Transit and at Rest)

데이터를 저장할 때와 전송할 때 모두 암호화를 적용합니다. 암호화 키를 안전하게 관리하고, 데이터 분류에 따른 보호 수준을 차등 적용합니다.

실천 항목 관련 챕터 ShopEasy 적용
저장 중 암호화 Ch.10 KMS, Ch.11 S3 보안, Ch.12 RDS 보안 S3 SSE-KMS, RDS 암호화, EBS 암호화
전송 중 암호화 Ch.14 CloudFront, Ch.12 RDS SSL/TLS HTTPS 강제, RDS SSL 연결 필수
키 관리 Ch.10 KMS CMK(고객 관리 키) 사용, 자동 로테이션
비밀 관리 Ch.13 Secrets Manager .env 파일에서 Secrets Manager로 전환

원칙 6: 사람의 데이터 직접 접근 최소화 (Minimize the Human Access to Data)

직접 데이터에 접근하는 것을 줄이고, 자동화된 도구와 프로세스를 통해 데이터를 처리합니다. 이를 통해 인적 오류와 내부자 위협 위험을 줄입니다.

실천 항목 관련 챕터 ShopEasy 적용
SSH 직접 접근 제거 Ch.09 네트워크 보안 실습 Session Manager로 SSH 대체, 22번 포트 차단
DB 직접 접근 제한 Ch.12 RDS 보안 IAM 인증 기반 DB 접근, 프라이빗 서브넷 격리
자동화된 배포 기초 과정 참조 CI/CD 파이프라인을 통한 자동 배포
S3 직접 접근 차단 Ch.11 S3 보안, Ch.14 CloudFront 보안 OAC를 통한 간접 접근만 허용

Before 아키텍처의 보안 취약점

식별된 주요 보안 취약점

계정 보안:
- IAM 사용자에 장기 액세스 키 사용
- MFA 미설정
- 루트 계정으로 일상 작업 수행
- 과도한 IAM 권한 (AdministratorAccess 사용)

네트워크 보안:
- EC2가 퍼블릭 서브넷에 위치
- SSH(22) 포트가 0.0.0.0/0으로 개방
- NACL 미설정 (기본값 사용)
- VPC Flow Logs 비활성화

데이터 보안:
- RDS 암호화 미적용
- DB 비밀번호가 .env 파일에 하드코딩
- S3 버킷 퍼블릭 접근 가능
- SSL/TLS 미강제 (HTTP 허용)

모니터링 & 대응:
- CloudTrail 미설정
- GuardDuty 미활성화
- 보안 알림 체계 없음
- 사고 대응 계획 없음

보안 아키텍처의 핵심 변화

After 아키텍처의 가장 중요한 변화는 "심층 방어"입니다. 단일 보안 계층이 실패하더라도 다른 계층이 보호를 제공합니다. WAF가 뚫려도 보안그룹이, 보안그룹이 뚫려도 NACL이, 네트워크가 뚫려도 암호화가 데이터를 보호합니다. 그리고 이 모든 것이 모니터링되고 자동 대응됩니다.

보안 성숙도 변화

Before 상태의 ShopEasy는 보안 성숙도 1단계("초기/임시" 수준)에 해당했습니다. After 상태는 3단계("정의됨/관리됨" 수준)에 도달했으며, 자동화와 지속적 개선을 통해 4단계("정량적 관리" 수준)로 발전할 수 있는 기반을 갖추었습니다.

과정을 마치며

32시간의 AWS 클라우드 보안 과정을 완료하셨습니다. 기초 과정에서 구축한 ShopEasy에 체계적인 보안 계층을 적용하는 여정을 함께 해주셨습니다.

기억하세요 -- 보안은 목적지가 아니라 여정입니다. 이 과정에서 배운 원칙과 도구들은 출발점입니다. 새로운 위협이 등장하고, 새로운 서비스가 출시되며, 규정이 변화함에 따라 보안 아키텍처도 지속적으로 발전시켜야 합니다.

Well-Architected Framework의 보안 필러 6가지 원칙을 항상 기억하고, 심층 방어 전략으로 모든 계층에 보안을 적용하며, 자동화를 통해 일관되고 신속한 보안 운영을 실현하세요.

AWS 보안 핵심 원칙 요약

1) 최소 권한: 필요한 만큼만 권한을 부여하세요
2) 심층 방어: 모든 계층에 보안을 적용하세요
3) 암호화: 저장 중이든 전송 중이든 데이터를 암호화하세요
4) 추적 가능성: 모든 활동을 기록하고 모니터링하세요
5) 자동화: 보안을 자동화하여 일관성과 속도를 확보하세요
6) 지속적 개선: 보안은 한 번 설정하고 끝나는 것이 아닙니다
