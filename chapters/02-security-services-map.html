Chapter 02

AWS 보안 서비스 전체 맵 & 이커머스 보안 위협 분석

학습 목표

  • AWS 보안 서비스를 5대 카테고리로 분류하고 각 서비스의 역할을 설명할 수 있다
  • 자격증명, 네트워크, 데이터 보호, 탐지, 사고 대응 서비스의 핵심 기능을 이해한다
  • ShopEasy 이커머스 아키텍처의 현재 보안 취약점을 식별할 수 있다
  • 보안 서비스 적용의 우선순위와 로드맵을 수립할 수 있다
  • 각 보안 서비스가 이 과정의 어느 챕터에서 다뤄지는지 파악한다

1. AWS 보안 서비스 5대 카테고리

AWS는 200개 이상의 서비스를 제공하며, 그 중 보안 관련 서비스만 30개가 넘습니다. 이 서비스들을 효과적으로 이해하려면 5대 카테고리로 분류하여 전체 그림을 먼저 파악하는 것이 중요합니다.

이 5가지 카테고리는 보안의 전체 라이프사이클을 나타냅니다: 예방(1,2,3) → 탐지(4) → 대응(5). 완전한 보안 체계를 갖추려면 5가지 카테고리 모두를 커버해야 합니다. 하나라도 빠지면 보안 사각지대가 생깁니다.

2. 자격증명 & 접근 관리 서비스

클라우드 보안의 첫 번째이자 가장 중요한 카테고리입니다. Chapter 01에서 배운 것처럼 "Identity is the New Perimeter", 즉 자격증명이 새로운 보안 경계입니다.

2.1 AWS IAM (Identity and Access Management)

AWS의 모든 접근 제어의 기반이 되는 서비스입니다. 사용자, 그룹, 역할, 정책을 통해 "누가, 어떤 리소스에, 어떤 작업을, 어떤 조건에서" 할 수 있는지를 정의합니다.

IAM 구성 요소 설명 ShopEasy 예시
사용자 (User) AWS 콘솔/CLI 접근을 위한 개별 계정 개발자, 운영자 각각 별도 IAM 사용자
그룹 (Group) 사용자를 묶어 동일 정책을 적용 Developers, Operations, ReadOnly 그룹
역할 (Role) 서비스나 사용자가 임시로 사용하는 권한 묶음 EC2 인스턴스 프로파일, Lambda 실행 역할
정책 (Policy) JSON으로 정의된 권한 규칙 S3 읽기 전용, RDS 접근 허용 정책
IAM 핵심 주의사항

루트 계정은 일상 작업에 절대 사용하지 마세요. MFA를 설정하고 금고에 넣어두세요. Access Key는 가능하면 사용하지 말고, IAM 역할과 임시 자격증명을 활용하세요. 이 내용은 Chapter 03~06에서 상세히 다룹니다.

2.2 AWS Organizations

여러 AWS 계정을 중앙에서 관리하는 서비스입니다. 서비스 제어 정책(SCP)을 통해 하위 계정에서 할 수 있는 작업의 최대 범위를 제한할 수 있습니다.

  • OU (Organizational Unit): 계정을 부서/역할별로 그룹화
  • SCP: 특정 OU나 계정에 적용하는 허용/거부 정책
  • 통합 결제: 모든 계정의 비용을 하나로 관리

2.3 AWS IAM Identity Center (구 AWS SSO)

여러 AWS 계정과 비즈니스 애플리케이션에 싱글 사인온(SSO)을 제공합니다. Active Directory, Okta 등 외부 IdP(Identity Provider)와 연동하여 중앙 집중식 인증을 구현합니다.

3. 네트워크 보안 서비스

네트워크 보안은 심층 방어의 핵심 계층입니다. VPC를 기반으로 여러 계층의 네트워크 방어를 구성합니다.

3.1 VPC (Virtual Private Cloud) 보안 구성 요소

구성 요소 작동 수준 특징 ShopEasy 활용
보안그룹 (SG) 인스턴스 레벨 상태 유지(Stateful), 허용 규칙만 설정 EC2, RDS, ALB 각각에 최소 포트만 개방

3.2 AWS WAF (Web Application Firewall)

웹 애플리케이션을 SQL Injection, XSS(Cross-Site Scripting), Bot 등의 공격으로부터 보호합니다. CloudFront, ALB, API Gateway에 연결하여 사용합니다.

  • Web ACL: 규칙의 모음. CloudFront/ALB에 연결
  • 관리형 규칙 (Managed Rules): AWS 및 마켓플레이스에서 제공하는 사전 정의된 규칙 세트
  • 커스텀 규칙: IP 기반, 요청 속도 기반, 문자열 매칭 등 직접 정의
  • Rate-based Rule: 특정 IP에서 5분간 요청 수 제한 (DDoS 완화)

3.3 AWS Shield

DDoS(분산 서비스 거부) 공격 방어 서비스입니다.

구분 Shield Standard Shield Advanced

3.4 AWS Firewall Manager

Organizations 환경에서 여러 계정의 WAF 규칙, Shield Advanced, 보안그룹을 중앙에서 일괄 관리합니다. 새로 생성되는 리소스에도 자동으로 보안 정책이 적용됩니다.

4. 데이터 보호 서비스

데이터는 기업의 가장 중요한 자산입니다. AWS는 데이터를 전송 중(in-transit)과 저장 시(at-rest) 모두 보호할 수 있는 서비스를 제공합니다.

4.1 AWS KMS (Key Management Service)

암호화 키를 생성, 저장, 관리하는 핵심 서비스입니다. 대부분의 AWS 서비스(S3, EBS, RDS, DynamoDB 등)와 통합되어 클릭 한 번으로 암호화를 활성화할 수 있습니다.

  • AWS 관리형 키: AWS가 자동 생성/관리. 무료
  • 고객 관리형 키 (CMK): 고객이 생성/관리. 키 정책, 로테이션 직접 제어
  • 봉투 암호화 (Envelope Encryption): 데이터 키를 마스터 키로 감싸는 효율적 암호화 방식

4.2 AWS CloudHSM

전용 하드웨어 보안 모듈(HSM)을 제공합니다. FIPS 140-2 Level 3 인증을 받은 하드웨어에서 키를 관리하며, 금융, 의료 등 엄격한 규제 요구사항이 있는 경우 사용합니다. KMS와 달리 고객이 키에 대한 완전한 통제권을 가집니다.

4.3 AWS Secrets Manager

DB 비밀번호, API 키, OAuth 토큰 등 비밀 정보를 안전하게 저장하고 자동 교체합니다.

  • 비밀번호를 코드나 .env 파일에 하드코딩하지 않아도 됨
  • Lambda를 활용한 자동 로테이션 (30/60/90일 주기)
  • RDS, Redshift, DocumentDB와의 기본 통합
  • KMS로 저장 시 암호화
Parameter Store vs Secrets Manager

SSM Parameter Store는 설정값 저장에 적합하고 무료(Standard 티어). Secrets Manager는 비밀번호 자동 교체가 필요할 때 사용하며 유료(비밀당 $0.40/월). ShopEasy에서는 DB 비밀번호는 Secrets Manager, 일반 설정은 Parameter Store를 사용합니다 (Chapter 13).

4.4 AWS Certificate Manager (ACM)

SSL/TLS 인증서를 무료로 발급, 관리, 갱신합니다. CloudFront, ALB, API Gateway에 연결하여 HTTPS를 쉽게 적용합니다. 인증서 만료 걱정 없이 자동 갱신됩니다.

4.5 Amazon Macie

머신러닝을 활용하여 S3 버킷에서 개인식별정보(PII), 금융 데이터 등 민감 데이터를 자동 탐지합니다. 데이터 분류(Data Classification)와 규제 준수에 활용됩니다.

5. 탐지 & 모니터링 서비스

예방이 아무리 완벽해도 보안 사고는 발생할 수 있습니다. 핵심은 얼마나 빨리 탐지하고 대응하느냐입니다. AWS의 탐지 서비스들은 서로 연동되어 포괄적인 보안 모니터링 체계를 구성합니다.

5.1 AWS CloudTrail

AWS 계정의 모든 API 호출을 기록하는 감사(Audit) 서비스입니다. "누가, 언제, 어떤 API를, 어디서 호출했는가"를 추적합니다.

  • 관리 이벤트: 리소스 생성/삭제/수정 (예: EC2 시작, IAM 정책 변경)
  • 데이터 이벤트: 리소스 내부 활동 (예: S3 객체 읽기/쓰기, Lambda 호출)
  • 인사이트 이벤트: 비정상적인 API 호출 패턴 탐지
CloudTrail은 보안의 기본 중 기본

CloudTrail을 활성화하지 않으면 보안 사고가 발생해도 원인 분석이 불가능합니다. "CCTV 없이 건물을 운영하는 것"과 같습니다. 반드시 첫 번째로 활성화해야 하는 서비스입니다 (Chapter 17).

5.2 AWS Config

AWS 리소스의 구성 변경 이력을 추적하고, 규칙 기반으로 규정 준수를 자동 확인합니다.

  • 구성 기록: "이 보안그룹이 언제, 누가 변경했는지" 이력 확인
  • Config Rules: "S3 버킷에 퍼블릭 읽기가 허용되면 비준수" 같은 규칙 자동 검사
  • 적합성 팩: 여러 규칙을 묶은 보안 템플릿 (PCI DSS, CIS Benchmark 등)
  • 자동 교정: 비준수 리소스를 SSM Automation으로 자동 수정

5.3 Amazon GuardDuty

머신러닝 기반의 위협 탐지 서비스입니다. CloudTrail, VPC Flow Logs, DNS Logs를 분석하여 의심스러운 활동을 자동으로 탐지합니다.

  • 암호화폐 채굴 탐지
  • IAM 자격증명 도용 탐지
  • 비정상적인 API 호출 패턴
  • 알려진 악성 IP/도메인과의 통신 탐지
  • S3 데이터 유출 시도 탐지

5.4 AWS Security Hub

여러 보안 서비스의 결과를 하나의 대시보드에 통합하고, 보안 점수를 제공합니다.

카테고리별 취약점 상세 분석

취약점 1: 자격증명 & 접근 관리

루트 계정으로 일상 작업 수행 - 루트 계정은 모든 권한을 가지므로 탈취 시 전체 인프라 장악 가능
MFA 미설정 - 비밀번호만으로 계정 접근 가능. 브루트포스/피싱 공격에 취약
Access Key 하드코딩 - EC2 인스턴스의 .env 파일이나 코드에 영구 Access Key가 저장됨
IAM 정책 과도 권한 - AdministratorAccess나 * 와일드카드 정책 사용

취약점 2: 네트워크 보안

SSH 포트 전체 개방 - 보안그룹에서 22번 포트가 0.0.0.0/0으로 열려 있어 전 세계에서 SSH 접속 시도 가능
WAF 미적용 - SQL Injection, XSS 등 웹 공격에 무방비
NACL 기본 설정 - 모든 인바운드/아웃바운드 트래픽 허용
VPC Flow Logs 비활성화 - 네트워크 트래픽 분석 불가

취약점 3: 데이터 보호

RDS 암호화 미적용 - 디스크에 데이터가 평문으로 저장됨
DB 비밀번호 .env 파일 저장 - 파일 유출 시 DB 접근 가능
S3 버킷 암호화 미적용 - 상품 이미지가 암호화 없이 저장됨
전송 중 암호화 미완성 - 일부 구간에서 HTTP 통신

취약점 4: 탐지 & 모니터링

CloudTrail 미설정 - API 호출 기록 없음. 보안 사고 시 원인 분석 불가
Config 미설정 - 리소스 구성 변경 추적 불가
GuardDuty 미활성화 - 위협 탐지 체계 없음
보안 알림 없음 - 비정상 활동 발생 시 통보 체계 부재

비유: 현재 ShopEasy는...

문을 열어둔 채로 영업하면서, CCTV도 없고, 금고 자물쇠도 안 잠그고, 직원 전원에게 마스터키를 나눠준 가게와 같습니다. 기능적으로는 작동하지만, 보안 사고가 발생하는 것은 시간문제입니다.

왜 이 순서인가?

IAM이 가장 먼저인 이유: 자격증명이 탈취되면 네트워크 보안이나 암호화가 무의미해집니다. 공격자가 관리자 권한을 가지면 보안그룹을 열고, 암호화를 해제하고, 로그를 삭제할 수 있기 때문입니다.

탐지가 마지막인 이유: 예방 조치가 먼저 갖춰진 상태에서 모니터링을 시작해야 불필요한 노이즈 없이 실제 위협에 집중할 수 있습니다.
